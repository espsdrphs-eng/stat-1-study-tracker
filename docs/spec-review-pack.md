# 統計検定1級 Study Tracker 仕様レビュー用パック

この文書は、統計検定1級・統計数理の学習管理アプリを、ChatGPTなどに仕様レビューしてもらうための説明資料です。

個人の学習履歴そのものではなく、アプリの目的・設計思想・データの扱い・学習フロー・判定ルールをまとめています。

## 1. アプリの目的

このアプリは、統計検定1級・統計数理の学習を、次のサイクルで効率よく回すためのローカルWebアプリです。

1. 今日やる問題を決める
2. iPad / GoodNotes / 解答シートで答案を書く
3. ChatGPTに答案を採点してもらう
4. ChatGPTの採点結果YAMLをアプリに貼り付ける
5. アプリが問題ID・評価・K/W/N/C・復習日・弱点傾向を更新する
6. 次回復習で、前回の弱点を踏まえて必要最小限の復習をする

単なるToDo管理ではなく、「問題 → 採点 → フィードバック → 復習 → 弱点傾向」の循環を作ることを目的にしています。

## 2. 利用環境と制約

- 主な利用端末は iPad。
- PCなしでも運用できることを重視する。
- アプリはPWAとしてiPadのホーム画面から使う想定。
- OpenAI APIは使わない。
- ChatGPT採点結果はユーザーがコピー＆ペーストで取り込む。
- 問題PDFや模範解答PDFをアプリが自動OCR解析することは、現時点では狙わない。
- 高度なAI自動採点・通知・クラウド同期は後回し。

## 3. セキュリティとデータ保存方針

- アプリ本体はGitHub Pages等で配信される。
- 学習記録・復習予定・弱点ノート・問題マスターは、基本的にブラウザ内のIndexedDBに保存する。
- サーバーDBは使わない。
- ユーザーの学習記録は外部サーバーへ送信しない。
- バックアップはJSONエクスポートで行う。
- 模範解答PDFを使う場合も、公開リポジトリや `public` フォルダに置くのではなく、端末内保存を原則とする。

## 4. 中心概念

### S問題

S問題は土台確認・復旧用です。

- 通常ロードマップの主役ではない。
- A問題や過去問でKが出たときの戻り先。
- 30日以上触っていない重要Sの忘却確認にも使う。
- 復習は短時間のチェック、骨格確認、復旧が中心。

### A問題

A問題は新規演習と復習の主役です。

- S問題の型を少し崩された問題への耐性を作る。
- K/W/N/Cに応じて復習間隔と復習方法を変える。
- 初回、復習、主要計算だけ、フル答案を分けて扱う。

### 過去問

過去問は本番選題力と答案化力の確認です。

- 5問から3問を選ぶスキャン練習。
- 90分で答案化する練習。
- 落とした型からA/S問題へ戻る。
- 初期段階では白本のS/Aを優先し、過去問比率は後半で上げる。

## 5. 問題IDと問題マスター

問題は必ず一意の `problem_id` で管理します。

例：

- `WB-6-A-05`: 白本 第6章 A問5
- `WB-6-S-21`: 白本 第6章 S問21
- `PY-2025-Q1`: 過去問 2025年 問1

同じ問番号でも章やS/Aで意味が変わるため、表示名ではなく `problem_id` を正本にします。

### problem_master / answer_index

アプリ内の正本はPDFではなく、ユーザー確認済みのJSONです。

- `problem_master.json`: 問題ID、表示名、章、S/A/過去問種別、テーマ、キーワード、関連問題を管理する。
- `answer_index.json`: 模範解答の有無、PDF上の位置、要約、キーワードを管理する。
- `aliases.json`: 表記ゆれを補正する。

GPT取り込み時は、GPTが出したテーマよりも `problem_master` を優先します。

例：

- `WB-6-S-01`: 指数型分布族・自然母数・期待値母数
- `WB-6-S-04`: U(0,θ)、十分統計量、不偏推定量、MSE、MLE

GPTの出力テーマが問題マスターと矛盾する場合は、アプリ側で補正し、元のGPTテーマは `raw_gpt_theme` として残します。

## 6. 学習モードの定義

### check

思い出せるかを確認する軽い復習です。

- C、none、安定S問題に使う。
- 型、初手、今見る量、注意点だけ確認する。
- フル答案は不要。

### skeleton

答案の設計図を作れるか確認する復習です。

- K、N、初見A問題の骨格確認、過去問スキャン後の補修に使う。
- 方針、出発式、今見る量、条件、道具、流れ、最後に示すことを書く。
- 最終式や完成答案までは求めない。

骨格で求めるもの：

- 方針・入口
- 出発式
- 今見る量
- 先に確認すること
- 使う道具
- 解答の流れ
- 最後に示すこと
- ここから先は計算

骨格で求めないもの：

- MLEの最終式を完全に出す
- 積分や和を最後まで計算する
- CR下限との一致を式で最後まで示す
- 信頼区間の具体的な端点まで出す

### main_calc

落とした計算だけ直す復習です。

- W、計算・積分・和の変形・場合分けで落ちたときに使う。
- 問題全体を解き直さない。
- 該当作業だけ再演習する。

### full

本番答案として書けるか確認するモードです。

- 重要A問題の初回フル答案、過去問90分演習、本番シミュレーションに使う。
- 出発式、途中式、条件、計算、結論まで書く。

## 7. K/W/N/Cの意味

- K: 型・方針・入口・主役の統計量・使う定理が崩れている。
- W: 計算・式変形・積分・和・場合分けなどの作業で落としている。
- N: ノート不足・説明不足・再現性不足。分かったつもりだが答案として再現できない。
- C: 符号・係数・範囲・条件確認などのケアレス。
- none: 大きな問題なし。

## 8. 復習間隔と復習方法

復習日は、GPT本文ではなくアプリ側のルールを正とします。

- Kあり: 1日後
- Nあり: 2日後
- Wあり: 3日後
- Cあり: 7日後
- none: 14日後
- ◎が2回連続: 30日後
- ◎が3回連続: 完了候補

複数ある場合は最短を採用します。

例：

- K + N: 1日後
- W + N: 2日後
- W + C: 3日後

復習方法の自動推定：

- Kあり: skeleton
- Nあり: skeleton
- Wあり: main_calc
- Cあり: check
- none: check

## 9. GPT回答取り込み

GPT採点結果の末尾に `study_update` または `study_updates` のYAMLを付けてもらい、アプリに貼り付けます。

### 保存に最低限必要な項目

- `problem_id`
- `date`
- `task_origin`
- `mode`
- `actual_minutes`
- `mark`
- `score_numeric`
- `error_types`
- `primary_error_type`
- `review_after_days`
- `next_action`

### 自動補完可能な項目

- `display_label`
- `main_theme`
- `themes`
- `review_method`
- `score_text`
- `error_point`
- `weak_notes`
- `linked_s_problems`
- `linked_past_exams`
- `s_check_suggestions`

`problem_master` に `problem_id` がある場合、表示名・テーマ・問題型・キーワードは問題マスターから補完します。

### 旧キーの自動変換

旧形式のYAMLも読めるようにします。

- `time_minutes` → `actual_minutes`
- `score_label` → `score_text`
- `reference_closed_reproduction` → `after_reference_reproduced`

元のキーは `raw_import_data` として残します。

### none判定の扱い

`error_types: ["none"]` または `primary_error_type: "none"` の場合：

- `error_point` が空欄でも保存可能。
- 表示上は「大きな問題なし」に補完してよい。
- `weak_notes` が空でも保存可能。

### problem_id 判定

問題IDの優先順位：

1. YAML内の `study_update.problem_id`
2. ユーザーが問題マスター欄で選択した `problem_id`
3. `problem_aliases` による補正
4. YAMLがない場合のみ、本文から候補推定

YAML内に明確な `problem_id` がある場合、本文中の「認識する表記例」などに含まれる別IDは候補にしません。

無視対象の見出し例：

- 認識する表記例
- 安定した採点用プロンプト
- プロンプト本文
- 例
- example
- template
- 使い方
- 入力例
- 出力例

### next_action の制約

`next_action` には「何をするか」だけを書き、日付や復習間隔を書かない方針です。

悪い例：

- 7日後にWB-6-S-01を骨格だけ再確認する。

良い例：

- WB-6-S-01を骨格だけ再確認し、各分布について母数と期待値の対応表を白紙で書く。

日付表現が含まれる場合は、保存前に警告または自動除去します。

## 10. task_origin

復習タスクには起源を持たせます。

- `first_attempt`: 初めて解く問題。
- `review_attempt`: 過去に解いた問題の復習。
- `linked_s_check`: A問題や過去問に付随する関連S確認。
- `related_drill`: 弱点補修用の派生課題。
- `past_exam_followup`: 過去問で落とした型の補修。

### first_attempt

前回ミスや前回反省点は表示しません。

表示するもの：

- この問題は初回です
- 今回の目的
- 初回答案で見るポイント
- 初回採点プロンプト

### review_attempt

前回ミス、修正テーマ、修正ルール例を扱います。

### linked_s_check

関連S確認は、そのS問題を「前回解いた問題」として扱いません。

表示例：

この問題自体は初回確認です。元問題の弱点から作られた関連確認です。

## 11. 復習画面のネタバレ防止

復習前に答えが見えすぎると学習効果が落ちるため、表示を段階化しています。

### 初期表示

答えを含まない情報だけ表示します。

- 今日の一手
- 今回の狙い
- 修正テーマ
- 今回の入口
- 1行ヒント
- 復習方法
- 使用シート
- 必要時間
- 最小完了条件
- 参照ルール

### 折りたたみ表示

ボタンを押したときだけ表示します。

- 前回ミスの詳細
- 修正ルール例
- 保存済みGPT解説
- 模範解答要約
- 公式解答

### 復習後表示

復習結果保存後に詳細を見てもよいものです。

- 前回ミスの詳細
- 修正ルール例
- 弱点ノート候補
- 関連S/A問題
- 次回課題

## 12. 修正テーマ・今回の入口・1行ヒント

抽象的すぎる復習指示は避け、問題固有語を入れます。

### 修正テーマ

形式：

【対象】について、【作業】を【注意点】つきで再現する。

例：

- 最大統計量について、分布関数から密度を出す作業を、定義域つきで再現する。
- 対数尤度について、微分して最尤推定量を出す作業を、最大化条件つきで再現する。
- AICについて、最大対数尤度と自由パラメータ数を分けて確認する。

### 今回の入口

どこから始めるかを示します。

例：

- 最大値がある値以下になる確率を、全標本がその値以下になる確率として書く。

### 1行ヒント

最終式や答えではなく、入口だけを具体化します。

例：

- まず最大統計量の分布関数を書き、そこから密度を出す。
- まず対数尤度を書き、パラメータを含む項だけを残して微分する。

## 13. 参照ルール

参照段階は2種類に分けます。

- `allowed_reference_level`: その復習タスクで許可される参照段階。
- `actual_reference_level`: 実際に見た最も深い参照段階。

参照段階：

0. `none`: 見ていない
1. `one_line_hint`: 1行ヒント
2. `previous_mistake`: 前回ミス
3. `saved_gpt_feedback`: 保存済みGPT解説
4. `official_answer`: 公式解答
5. `external_reference`: 外部参照

NやWの補修復習では、前回ミスを見ることを復習の一部として許可できます。

重要なのは「見たかどうか」だけではなく、「表示を隠してから白紙で再現したか」です。

参照内容は開閉できます。一度見た参照段階は記録に残りますが、表示を隠してから白紙で再現します。

## 14. 今日やること

今日やることは `today_plan_snapshot` を正本にします。

日付が変わった最初の表示時、今日の予定をスナップショットとして保存し、その日中は勝手に再計算しません。

変わるのはユーザー操作があったときだけです。

- 解答済みにする
- 明日に送る
- 3日後へ送る
- 週末へ送る
- 期限なしにする
- 今日やるに戻す
- 今日の予定を再計算する

### 時間表示

時間表示は次のように分けます。

- `start_of_day_planned_minutes`: 今日開始時点の総予定時間
- `completed_minutes_today`: 今日完了登録した実績時間
- `active_remaining_minutes`: 今日これから実際にやる予定時間
- `postpone_candidate_minutes`: 先送り候補の時間
- `postponed_minutes_today`: 実際に今日から先送り済みの時間
- `target_minutes_today`: 1日の目標時間

先送り候補は、今日の実行予定には含めません。

警告判定は、

`completed_minutes_today + active_remaining_minutes`

が `target_minutes_today` を大きく超える場合に出します。

## 15. 今日やることの仕分け

今日の課題は次に分類します。

- 必ずやる
- 余裕があれば
- 先送り候補
- 完了済み
- すべて

優先順位：

1. Kを含む
2. Nを含む
3. 過去問に直結する重要A問題
4. 期限切れ日数が大きい
5. Wを含む
6. Cのみ
7. none
8. S問題メンテ
9. 緊急性の低い linked_s_check

KやNを先送りする場合は警告します。

## 16. ダッシュボード

ダッシュボードの「次の課題」は、今日やること画面と同じ active task queue を参照します。

優先順位：

1. 今日やることの「必ずやる」の先頭
2. 「必ずやる」が空なら「余裕があれば」の先頭
3. それも空なら「復習待ち」の最優先
4. それも空なら「今日は完了」

ダッシュボードには表示元も出します。

例：

- 表示元：今日やること > 必ずやる > 1番目
- 表示元：復習予定 > 期限切れ > K優先

## 17. 弱点分析

弱点ノートは、単なるミス一覧ではなく、苦手傾向を見つけるために使います。

表示方針：

- K/W/N/Cの傾向
- 章ごとの弱点
- テーマごとの再発
- 最近増えている弱点
- 復習すべきテーマ

弱点の復習は、アプリ内で細かく問題を出すよりも、ChatGPTにクイズ形式で復習してもらいやすい形のプロンプトを出す方針です。

弱点分析・対策候補は編集・削除できます。

## 18. 過去問の扱い

過去問は、最初から主軸にしすぎず、白本S/Aで土台を作った後に比率を上げます。

残り日数の目安に応じて、次のように考えます。

- 残り140〜100日: S再固定 + A+着手
- 残り99〜60日: 第5章A+、第7章A+、第3章A+を補強
- 残り59〜25日: 過去問主軸
- 残り24〜0日: 弱点補修 + 本番シミュレーション

過去問モード：

- 5問スキャン
- 90分答案
- 単問補修
- 過去問フォローアップ

## 19. 初回採点プロンプトと復習採点プロンプト

初回問題と復習問題では、GPTに渡すプロンプトを分けます。

### 初回採点プロンプト

- 初めて解いた問題の到達度診断。
- 前回ミスや復習履歴は使わない。
- `problem_id` は指定値を必ず使わせる。
- 方針・出発式・主要計算・条件・結論・再現可能性を評価する。

### 復習採点プロンプト

- 前回の弱点が改善されたかを見る。
- ただし復習方法に応じて要求範囲を変える。
- skeletonなら最終式までは要求しない。
- main_calcなら該当計算だけを見る。
- fullなら本番答案として見る。

## 20. GPTレビューで見てほしい論点

以下を重点的にレビューしてほしいです。

1. この学習サイクルは、統計検定1級の学習効率を高める設計になっているか。
2. K/W/N/Cと復習間隔のルールは妥当か。
3. skeleton / main_calc / full の分離は適切か。
4. GPT採点をコピペ運用する設計に、破綻しやすい点はないか。
5. iPadのみの運用として分かりやすいか。
6. 今日やること、復習予定、ダッシュボードの情報量は過不足ないか。
7. problem_master / answer_index を正本にする設計は堅いか。
8. ネタバレ防止と復習効率のバランスはよいか。
9. 弱点分析は、学習行動につながる形になっているか。
10. 残り日数を考えた進捗確認項目として、過去問やA+の扱いは妥当か。

## 21. GPTレビューで前提にしてほしいこと

次の提案は、現時点では優先度を下げてください。

- OpenAI API連携
- OCR自動解析
- PDF全文自動解析
- クラウドDB化
- 通知機能
- 複雑なAI自動採点

まずは、iPadだけで毎日破綻なく回せること、そして「問題 → 採点 → 復習」が軽く回ることを重視します。
